\chapter{System Implementation}

\section{Implementation Methodology, Algorithms, and Technology Stack}
The implementation of the proposed churn prediction system is based on a standard machine learning workflow adapted for structured customer data. Although the present report emphasizes documentation rather than source code, the methodology is described in a form that can be directly translated into practice using Python and widely used data science libraries. The implementation begins with data collection and loading. Customer records may be stored in CSV files, relational databases, or internal application logs. These records are first imported into a tabular data structure such as a pandas DataFrame, where inspection and preprocessing can be performed efficiently. Typical early checks include identifying null values, verifying data types, detecting duplicates, and examining class distribution in the target variable.

Once the data is loaded, preprocessing becomes the most important implementation stage. In any realistic churn dataset, a number of columns will require preparation before training can begin. Customer identifiers such as \texttt{CustomerID} are removed because they do not contribute predictive value and may encourage spurious pattern learning. Numeric attributes are examined for missing values and unusual distributions. Depending on the variable, mean or median imputation may be used, while severe outliers may be clipped or winsorized to reduce instability. Categorical fields such as region, gender, plan type, and active status are transformed through one-hot encoding so that machine learning algorithms can interpret them properly. Numerical features may also be standardized, especially when models like Logistic Regression are included in the comparison set.

Feature engineering extends the implementation beyond basic cleaning. In churn prediction, raw columns rarely capture customer behavior completely. It is often more informative to derive variables such as transaction rate, complaint frequency, balance-to-tenure relationships, or onboarding period indicators. For example, a customer with ten transactions over two months may be significantly more engaged than a customer with the same transaction count over two years. Likewise, two complaints in the first month after joining may be more alarming than two complaints over several years. Feature engineering therefore converts operational data into business-behavior indicators that improve the model’s ability to detect attrition risk.

Model development in this project includes four algorithms: Logistic Regression, Decision Tree, Random Forest, and XGBoost. Logistic Regression is implemented as a baseline because it is simple, efficient, and interpretable. Decision Trees are useful because they model non-linearity and yield rules that can be visualized. Random Forest improves stability and reduces the overfitting tendency of a single tree by averaging many trees trained on sampled data. XGBoost is selected because it has demonstrated strong performance in many tabular classification tasks and is widely used in churn prediction studies \cite{imani2024,bhattacharjee2026}. Each model is trained on the prepared training set and evaluated first under default settings before hyperparameter tuning.

Hyperparameter tuning is implemented through cross-validated search. For Logistic Regression, the strength of regularization is an important factor. For Decision Tree and Random Forest, parameters such as maximum depth, minimum samples required for a split, and number of estimators influence both accuracy and generalization. Because churn datasets are usually imbalanced, implementation also includes either SMOTE-based oversampling or class weighting. This step is important because a model trained on heavily skewed data may learn to favor the majority non-churn class excessively. The training pipeline therefore separates the training and testing sets before any resampling is applied so that the test set remains an honest measure of performance. A typical workflow is to use an 80:20 split, perform oversampling only on the training portion, and then evaluate the trained models on the untouched test portion.

The technology stack for the project is intentionally practical and accessible. Python provides the main development environment. pandas and NumPy support data transformation and numerical computation. scikit-learn provides implementations of preprocessing pipelines, classical machine learning models, train-test splitting, cross-validation, and evaluation functions. XGBoost supplies the boosted ensemble classifier. matplotlib and seaborn can be used to create the confusion matrix, ROC curve, and feature importance charts. For deployment-oriented demonstration, a simple Flask-based API and dashboard can be added to expose predictions and display customer risk categories. This stack is appropriate for a final-year academic project because it is powerful enough for serious experimentation while remaining understandable and reproducible.
