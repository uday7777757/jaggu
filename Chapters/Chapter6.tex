\chapter{System Testing and Evaluation}

\section{Testing Strategy, Evaluation Metrics, and Validation Approach}
The testing and evaluation stage of a machine learning project is crucial because the quality of the final system cannot be judged by model training alone. A churn prediction solution may appear to work well during development but still fail to support reliable business decisions if testing is shallow, metrics are chosen poorly, or evaluation ignores the imbalance in the target class. In this project, testing is treated as a multi-level process that validates both the analytical correctness of the pipeline and the usefulness of the system outputs. The overall testing strategy includes unit-level checking of preprocessing operations, integration-level validation of the end-to-end workflow, and model-level evaluation on unseen data.

At the unit-testing level, individual operations in the machine learning pipeline should be verified separately. This includes checking whether missing-value imputation handles nulls correctly, whether categorical encoding produces the expected feature set, whether scaling is applied only to intended numerical fields, and whether the prediction function returns outputs in the required format. Such validation may seem technical, but it is essential because small mistakes in preprocessing can cause major downstream errors. For example, applying one-hot encoding inconsistently between training and testing data can produce shape mismatches or silently reduce model quality. Similarly, leakage of test information into preprocessing steps can lead to inflated performance estimates.

Integration testing focuses on the full workflow from data input to decision output. In the context of this project, an end-to-end test would verify that a customer dataset can be loaded, cleaned, transformed, scored by the trained model, and represented in a dashboard-friendly format without failure. Integration testing is especially useful when the system includes multiple components such as a stored trained model, a preprocessing pipeline, a scoring function, and a simple interface for displaying results. Even if each component works individually, their interaction may still fail unless tested together.

The most significant evaluation work, however, occurs at the model level. Since the project deals with customer churn prediction, it is not sufficient to report overall accuracy and declare success. In an imbalanced dataset, accuracy can be misleading because the majority non-churn class dominates the score. Suppose only 20 percent of the customers churn. A model that predicts every customer as retained could still achieve 80 percent accuracy while failing to identify any churner at all. This is why the literature recommends a broader evaluation framework including precision, recall, F1-score, ROC-AUC, and confusion matrix analysis \cite{systematicreview,imani2024,bhattacharjee2026}. Recall is especially important in churn analysis because it measures how many actual churners are correctly identified. Missing true churners can have direct financial cost, since those customers will not receive intervention in time.

The metrics used in this project are summarized in Table \ref{tab:evaluation_metrics_main}. Accuracy captures overall correctness, precision measures how reliable positive churn predictions are, recall measures how many true churners are detected, F1-score balances precision and recall, and ROC-AUC captures ranking quality across classification thresholds. Together, these metrics provide a more realistic understanding of model behavior.

\begin{table}[H]
\centering
\caption{Evaluation Metrics}
\begin{tabular}{|p{3cm}|p{8.5cm}|}
\hline
\textbf{Metric} & \textbf{Definition} \\ \hline
Accuracy & $(TP + TN) / (TP + TN + FP + FN)$ \\ \hline
Precision & $TP / (TP + FP)$ \\ \hline
Recall & $TP / (TP + FN)$ \\ \hline
F1-Score & $2 \times \frac{Precision \times Recall}{Precision + Recall}$ \\ \hline
ROC-AUC & Area under the receiver operating characteristic curve \\ \hline
\end{tabular}
\label{tab:evaluation_metrics_main}
\end{table}

To support fair comparison, the dataset is split into training and testing portions before modeling. Cross-validation on the training set is used during hyperparameter tuning so that the system does not overfit to a single split. The test set remains unseen until final model comparison. If oversampling methods such as SMOTE are used, they are applied only to the training data. This prevents target leakage and ensures that final evaluation reflects realistic deployment conditions. The confusion matrix provides an especially useful diagnostic view because it shows the trade-off between false positives and false negatives. In churn management, a moderate number of false positives may be acceptable if it helps capture more real churners, but this balance depends on business cost assumptions.

Overall, the testing strategy used in this project is designed to ensure that the proposed system is not only technically executable but also analytically trustworthy. By combining pipeline validation, end-to-end testing, multi-metric model evaluation, and confusion-matrix-based interpretation, the project creates a balanced framework for selecting a churn model that is appropriate for practical decision support.
