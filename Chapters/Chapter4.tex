\chapter{System Design}

\section{Architecture, Data Flow, UML Design, and Deployment View}
The design of the proposed churn prediction system is intended to support not only model training, but also practical use by business teams. For that reason, the system is organized as a layered pipeline with clear movement of data from raw source systems to end-user decision support. At the highest level, the architecture can be divided into four connected layers: the data layer, the processing layer, the service layer, and the presentation layer. The data layer represents source systems such as customer relationship management databases, transactional databases, and complaint or interaction logs. These systems contain heterogeneous customer information that must be extracted and consolidated before predictive modeling becomes possible.

The processing layer performs the most analytically important work. Once data is ingested, it enters the preprocessing module, where missing values are handled, inconsistent entries are corrected, outliers are reviewed, categorical variables are encoded, and numeric features are scaled where required. This layer also includes feature engineering, because raw operational fields often need to be converted into more useful predictive indicators. For example, the raw number of transactions alone may not be sufficiently informative unless interpreted relative to customer tenure. Similarly, complaints become more meaningful when viewed as a rate rather than a total count. After these transformations, the prepared dataset is passed into model training and tuning components. Here, multiple models are fitted and compared using cross-validation, and the best-performing model is selected for deployment.

Once a final model is chosen, it becomes part of the service layer. In a practical implementation, this layer would expose the model through an inference service or API. New customer records or scheduled customer batches could then be scored without retraining the model every time. The service layer may also include a notification or rules component that converts churn probabilities into business actions such as retention alerts, at-risk customer lists, or campaign triggers. The presentation layer sits above the service layer and translates model outputs into a dashboard that can be used by managers, analysts, or customer success teams. Instead of showing only binary churn predictions, the presentation layer is designed to display risk categories, customer-level details, and the most influential features behind the decision. This improves usability and encourages adoption by non-technical stakeholders.

From a data movement perspective, the same design can be understood through data flow diagrams. At the context level, the entire application behaves as a single churn prediction process that receives customer information and produces risk reports. At a more detailed level, this process can be decomposed into data collection, cleaning, feature engineering, model scoring, and reporting. These DFDs are useful because they show that the system is not merely a model but a connected decision-support pipeline. They also help clarify the boundary between data preparation and business output. UML diagrams complement the architecture and DFD views by showing internal structure and interactions. The class diagram identifies conceptual objects such as Customer, Account, Transaction, ChurnModel, and Dashboard. The sequence diagram describes the operational flow when data is submitted for scoring and results are returned to the user interface. The activity diagram captures the iterative nature of model development, where unsatisfactory evaluation can lead back to feature engineering and retraining. The deployment diagram illustrates how the system could be hosted on a cloud server with an application layer, database, and model service. These representations help convert a machine learning experiment into a complete software engineering artifact, which is a key requirement for an academic major project.
