\chapter{Testing and Evaluation}

\section{Comprehensive Testing Approach}

The testing strategy encompasses four complementary levels, following standard software quality
assurance practices:

\subsection{Unit Testing}

Unit tests validate individual functions and methods in isolation, using synthetic fixtures to
avoid dependency on real data. The \texttt{pytest} framework is used, with coverage reported via
\texttt{pytest-cov}. Key unit tests include:

\begin{itemize}
    \item \textbf{Imputation tests}: Verify that median imputation produces the expected output
          for numerical columns containing known missing values.
    \item \textbf{Feature engineering tests}: Assert that RFM values are correctly computed
          from controlled transaction histories with known ground-truth outputs.
    \item \textbf{SHAP attribution tests}: Confirm that the sum of SHAP values for each
          prediction equals the difference between the prediction and the model's expected output,
          satisfying the local accuracy property.
    \item \textbf{API endpoint tests}: Using FastAPI's \texttt{TestClient}, verify that
          \texttt{/health} returns a 200 status code and that \texttt{/score/batch} returns the
          correct JSON schema for a minimal valid payload.
\end{itemize}

Overall unit test coverage is 87\%.

\subsection{Integration Testing}

Integration tests execute the full pipeline end-to-end on a small held-out sample (500 customers)
and assert that:

\begin{itemize}
    \item The pipeline produces a prediction for every customer in the sample (no dropped rows).
    \item All predicted probabilities lie in the interval $[0, 1]$.
    \item The SHAP attribution DataFrame has the correct shape and column names.
    \item The API response for the batch endpoint contains all requested customer identifiers.
\end{itemize}

\subsection{Model Validation Testing}

Model validation tests operate on the full hold-out test set and assert performance thresholds:

\begin{itemize}
    \item ROC-AUC $\geq 0.90$.
    \item Recall $\geq 0.85$ at the operating threshold.
    \item No single feature SHAP magnitude exceeds 0.4, indicating no single-feature over-reliance.
\end{itemize}

These thresholds are codified as automated assertions in the CI/CD pipeline so that model updates
that degrade performance below acceptable levels are automatically rejected.

\subsection{Load Testing}

Load testing uses Locust to simulate concurrent API requests. The inference service is tested
under a simulated load of 500 concurrent users submitting batch scoring requests for 100 customers
each. Performance targets are:

\begin{itemize}
    \item Mean response time $\leq 200$ ms.
    \item 99th percentile response time $\leq 800$ ms.
    \item Zero failed requests under 300 concurrent users.
\end{itemize}

\section{Evaluation Metrics Framework}

\subsection{Primary Metrics}

Table~\ref{tab:metrics} provides formal definitions of the evaluation metrics used.

\begin{table}[H]
\centering
\caption{Evaluation Metrics Definitions and Formulae}
\label{tab:metrics}
\begin{tabular}{|l|p{5cm}|p{4.5cm}|}
\hline
\textbf{Metric} & \textbf{Definition} & \textbf{Formula} \\
\hline
Accuracy & Proportion of correct predictions & $\frac{TP + TN}{TP + TN + FP + FN}$ \\
\hline
Precision & Proportion of predicted churners that are true churners & $\frac{TP}{TP + FP}$ \\
\hline
Recall (Sensitivity) & Proportion of actual churners correctly identified & $\frac{TP}{TP + FN}$ \\
\hline
F1-Score & Harmonic mean of precision and recall & $\frac{2 \times P \times R}{P + R}$ \\
\hline
ROC-AUC & Area under the ROC curve & $\int_0^1 TPR\, d(FPR)$ \\
\hline
Specificity & Proportion of non-churners correctly identified & $\frac{TN}{TN + FP}$ \\
\hline
Top-Decile Lift & Churn rate in top 10\% / overall churn rate & $\frac{CR_{D1}}{CR_{total}}$ \\
\hline
KS Statistic & Max separation between churn and no-churn CDF & $\max|CDF_{churn} - CDF_{no-churn}|$ \\
\hline
\end{tabular}
\end{table}

\noindent
where $TP$, $TN$, $FP$, $FN$ denote True Positives, True Negatives, False Positives, and
False Negatives respectively; $P$ denotes Precision; $R$ denotes Recall; $CR_{D1}$ denotes
the churn rate in the top decile; and $CR_{total}$ denotes the overall population churn rate.

\subsection{Business-Oriented Metrics}

Beyond statistical performance metrics, the following business-oriented quantities are tracked:

\begin{itemize}
    \item \textbf{Expected Maximum Profit (EMP)}: The maximum profit achievable by targeting an
          optimal subset of scored customers, calculated as a function of customer lifetime value,
          campaign cost, and the predicted churn probability distribution.
    \item \textbf{Retention ROI}: The ratio of retained revenue (expected lifetime value of
          successfully retained customers) to campaign cost.
    \item \textbf{Coverage at recall 85\%}: The fraction of the customer base that must be
          contacted to achieve 85\% recall of churners.
\end{itemize}

\section{Visualisations and Analysis}

\subsection{ROC Curve Analysis}

The ROC curve plots the True Positive Rate (Recall) against the False Positive Rate
(1 -- Specificity) as the decision threshold varies from 0 to 1. The area under this curve
(AUC) for the tuned XGBoost model on the test set is 0.9247, indicating that the model
correctly ranks a randomly chosen churner above a randomly chosen non-churner in 92.47\% of
pairwise comparisons.

\subsection{Precision--Recall Curve}

The Precision--Recall (PR) curve is more informative than the ROC curve under significant class
imbalance. The average precision (area under the PR curve) for the tuned XGBoost model is 0.8134.
The operating threshold is selected as the point that maximises the F1-score, yielding a
threshold of 0.38 (lower than the default 0.5 due to class imbalance).

\subsection{Confusion Matrix}

At the selected threshold of 0.38 applied to the 10{,}000-customer test set (1{,}650 actual
churners, 8{,}350 non-churners):

\[
\text{Confusion Matrix} =
\begin{pmatrix}
TN = 7{,}475 & FP = 875 \\
FN = 197    & TP = 1{,}453
\end{pmatrix}
\]

This yields: Recall = 88.1\%, Precision = 62.4\%, Specificity = 89.5\%. Note that the precision
of 62.4\% is computed at the operating threshold of 0.38, which is chosen to maximise recall
rather than precision. The cross-validated precision reported in Table~\ref{tab:algocmp}
(Chapter~5) and Table~\ref{tab:finalresults} (Chapter~7) reflects the default threshold of 0.50
and is therefore higher (80.9\%); practitioners should select the operating threshold based on
the business cost ratio between false positives and false negatives.

\subsection{Lift and Gain Charts}

The top-decile lift of 4.7 indicates that targeting the 10\% of customers with the highest
predicted churn probabilities captures 47\% of all actual churners, a fourfold improvement over
random targeting. The KS statistic is 0.614, well above the commercial viability threshold of
0.40.
